Page Load, Title & Focus Protocol

V-1.1 GlobalRouteObserver is never mounted in App.tsx Critical ▼
File: client/src/App.tsx:30–55 WCAG: 2.4.3 Focus Order

App.tsx imports FocusManager (the context provider) and AccessibilityAnnouncer, but never imports or renders GlobalRouteObserver. The export exists at FocusManager.js:174 but is unused anywhere in the rendered tree.

// App.tsx — GlobalRouteObserver is absent from all imports and JSX
import FocusManager from './components/Accessibility/FocusManager'; // context only

// Inside <div id="app-root">:
<AccessibilityAnnouncer />
<FocusManager />         {/* provider only — no route observer */}
<Layout> ... </Layout>

// GlobalRouteObserver is exported from FocusManager.js line 174 but never used
Impact No focus reset, no h1 focus, and no route-change title sequencing happens on any route change in the entire application. Every WCAG 2.4.3 navigation requirement fails because of this single missing render.
V-1.2 document.title → h1 focus sequence is not guaranteed High ▼
File: FocusManager.js:200–209 WCAG: 2.4.2 Page Titled

focusPageHeadingOrMain() fires at 120ms / 420ms / 900ms after route change. <Helmet> title updates happen asynchronously and are not coordinated with these timers. There is no mechanism to verify document.title has been updated before focus moves to <h1>.

// FocusManager.js — focus timers run with no title-update check
const timers = [120, 420, 900].map((delay) =>
    setTimeout(attemptFocus, delay)   // no await on Helmet title update
);
Impact Screen readers may announce the previous page title simultaneously with the new h1 content, causing double-announce confusion.
V-1.3 <h1> elements lack tabIndex={-1} as a React prop High ▼
Files: EventDetail.js:100, LoginPage.js:210 WCAG: 2.4.3 Focus Order

The spec requires every view's <h1> to carry tabindex="-1" permanently. focusPageHeadingOrMain() patches this with setAttribute('tabindex', '-1') at runtime, but this DOM mutation is lost on any React re-render that replaces the element.

// EventDetail.js:100 — no tabIndex prop
<h1 style={{ textAlign: 'center', textTransform: 'uppercase' }}>
    {event.name}
</h1>

// LoginPage.js:210 — no tabIndex prop
<h1 className="login-title">
    {userType === 'jobseeker' ? 'Job Seeker Login' : 'Company & Staff Login'}
</h1>

// The fix in FocusManager.js:155 is fragile — DOM attribute lost on re-render
heading.setAttribute('tabindex', '-1');
heading.focus();
Impact Programmatic .focus() on an element without tabindex="-1" silently fails in most browsers. Screen readers receive no focus event and therefore no page-load signal.
V-1.4 Auth.css adds a visible outline to h1:focus — Focus Silence violation High ▼
File: Auth.css:548–556 Conflicts with: index.css:32–38 WCAG: 2.4.7 Focus Visible

index.css:32 correctly silences heading focus. However Auth.css loads after it and overrides with a visible blue outline + blue background fill on every auth page. The spec mandates outline: none for programmatic <h1> focus.

/* index.css:32 — correct silence rule */
h1:focus { outline: none; }

/* Auth.css:548 — VIOLATION: overrides and adds visible styling */
.auth-title:focus,
h1:focus {
    outline: 3px solid #3b82f6;             /* visible ring */
    outline-offset: 4px;
    background-color: rgba(59, 130, 246, 0.1); /* visible fill */
    border-radius: 4px;
    padding: 4px 8px;
    margin: -4px -8px;
}
Impact Screen reader users on Login / Register / Forgot Password pages see an unexpected blue halo flash on the heading when the page loads. Sighted keyboard users are confused by a non-interactive element receiving visible focus styling.
V-1.5 focusPageHeadingOrMain falls through to <h2> / <h3> Medium ▼
File: FocusManager.js:153–154 WCAG: 1.3.1 Info and Relationships, 2.4.3

The focus function uses a multi-element selector. If a route renders without an <h1>, focus silently lands on a subheading that is not the designated target and may lack tabindex="-1".

// FocusManager.js:153-154 — h2/h3 are not acceptable fallbacks
const heading = (mainArea || document).querySelector('h1, h2, h3');
Impact Any view that is missing an <h1> produces a silent failure — focus lands on the wrong element without any error. The root cause (missing h1) goes undetected.
V-1.6 Menu-initiated navigation receives zero focus management High ▼
File: FocusManager.js:199–206 WCAG: 2.4.3 Focus Order

When sidebar navigation is detected, the entire focus reset block is skipped. Screen reader users who navigate via the menu get no page-load signal — focus remains wherever it was in the previous view.

// FocusManager.js:200-206
const attemptFocus = () => {
    if (!menuInitiatedNav) {
        focusPageHeadingOrMain();   // skipped for sidebar navigation
    }
    // nothing else happens when menuInitiatedNav is true
};
Impact Screen reader users using the sidebar — the primary navigation path — receive no confirmation that a page change occurred. This fails WCAG 2.4.3 for the most common navigation pattern in the application.

Navigation & SPA Routing

V-3.1 Skip link is NOT the first focusable DOM element on the Login page High ▼
File: LoginPage.js:172–174 WCAG: 2.4.1 Bypass Blocks

PublicSiteHeader is rendered before the skip link in JSX. The header contains a focusable <Link> element, so keyboard Tab order hits the logo link first, then the skip link. The spec is unambiguous: the skip link must be the first focusable element in the DOM.

// LoginPage.js:172-174 — wrong order
<PublicSiteHeader />                                 {/* ← contains focusable <Link> */}
<a href="#main-content" className="skip-link">       {/* ← arrives SECOND in tab order */}
    Skip to main content
</a>
Impact Keyboard users must navigate through the header before they can access the skip link. The skip link's entire purpose is defeated when it isn't first.
V-3.2 Skip link is discoverable by Screen Reader Virtual Cursor Medium ▼
File: index.css:41–56 WCAG: 2.4.1 Bypass Blocks

The skip link uses position: absolute; top: -40px to hide it visually. This off-screen technique keeps the element in the accessibility tree, allowing virtual cursor (Up/Down arrow keys in JAWS/NVDA) to land on it at any time. The spec requires it to be skipped by the Screen Reader Virtual Cursor — accessible only via keyboard Tab.

/* index.css:41-56 */
.skip-link {
    position: absolute;
    top: -40px;    /* visually hidden but still in accessibility tree */
    left: 6px;
    z-index: 3000;
}
.skip-link:focus { top: 6px; }
/* ✗ No clip / clip-path to remove from virtual cursor flow */
Impact Virtual cursor users encounter the skip link unexpectedly while reading page content. The link announces "Skip to main content" at random points in the AT reading flow, causing confusion.
V-3.3 No id="app-root" focus reset on route change High ▼
File: FocusManager.js:152–168 WCAG: 2.4.3 Focus Order

The spec requires focus to move to id="app-root" (with tabindex="-1") on every route change, so the next Tab press lands on the skip link. focusPageHeadingOrMain() instead targets the <h1> directly, bypassing the established reset point. Additionally, since GlobalRouteObserver is not mounted (V-1.1), neither target receives focus.

// FocusManager.js:152 — targets h1 directly, not app-root
const focusPageHeadingOrMain = () => {
    const mainArea = document.getElementById('main-content') || ...;
    const heading = (mainArea || document).querySelector('h1, h2, h3');
    if (heading) {
        heading.focus();
        return true;
    }
    // app-root is never focused as a reset point
};
Impact After navigation, keyboard users cannot rely on Tab landing predictably on the skip link. The flow breaks the standard SPA navigation pattern.

The ExternalLink component is correctly implemented, but event descriptions are rendered via dangerouslySetInnerHTML. Any external links inside that HTML bypass the component entirely, lacking the external-link icon, the aria-label "(opens in new window)" suffix, and the aria-live announcement.

{/* Any <a target="_blank"> inside event.description has:
    ✗ No external-link SVG icon

Silent Failure Audit (Edge Cases)

V-7.1 useModalAriaHidden applies aria-hidden to #app-root — which contains the modal Critical ▼
File: useModalAriaHidden.js:15–28 WCAG: 1.3.1, 2.4.3 Focus Order

When a modal opens, the hook sets aria-hidden="true" on #app-root. Since modals are rendered as children of <Layout> inside #app-root, the modal DOM is inside the hidden subtree. This makes the modal completely invisible to screen readers — the exact opposite of the intended effect.

The correct ARIA modal pattern applies aria-hidden to sibling containers (header, nav, main) while the modal's container remains in a visible, non-hidden branch of the DOM tree.

// useModalAriaHidden.js:15-28
const appRoot = document.getElementById('app-root');
if (isOpen) {
    appRoot.setAttribute('aria-hidden', 'true');  // hides #app-root
}

// DOM tree — modal is INSIDE #app-root:
// #app-root [aria-hidden="true"]     ← hidden from screen reader
//   └─ <Layout>
//       └─ <Modal>                   ← modal is inside hidden node
//           └─ <dialog> content      ← screen reader cannot reach this
Critical Impact Every modal in the application that uses this hook is inaccessible to screen reader users. They cannot interact with modal content at all while it is open. This is a regression from the intended WCAG 2.4.3 compliance.

// CallControls.js:218-274
{showMoreOptions && (

V-7.3 CallControls removes the wrong event listener — Escape handler leaks High ▼
File: CallControls.js:44–55 WCAG: 2.1.1 Keyboard

The keydown listener is attached as an anonymous inline function, which cannot be referenced for removal. The cleanup function attempts to remove handleClickOutside from the keydown event — a handler that was never attached to keydown.
