Remaining Violations (4 Issues)

❌ REMAINING #1 — External Links Missing Full Accessibility Treatment
WCAG 2.4.4 Link Purpose WCAG 3.2.2 On Input Severity: High Files: 5+ components

All target="_blank" links must implement the complete 3-part pattern:

// Recommended: Create a reusable ExternalLink component
const ExternalLink = ({ href, children, label }) => {
  const { announce } = useAccessibilityAnnouncer();
  const handleClick = () => {
    announce('Opening external link. Leaving application.', 'polite');
  };
  return (
    <a
      href={href}
      target="_blank"
      rel="noopener noreferrer"
      aria-label={`${label} (opens in new window)`}
      onClick={handleClick}
    >
      {children}
      <svg aria-hidden="true" style={{color:'#595959'}} ...>↗</svg>
    </a>
  );
};

Affected files to update: RegisterPage.js, BoothQueueEntry.js, JobSeekerManagement.js, EventDetail.js, JobSeekerProfileModal.js

⚠️ REMAINING #2 — Banner Missing on Public Routes
WCAG 1.3.1 Info & Relationships Severity: Medium Files: LoginPage.js, RegisterPage.js, HomePage.js

The <header role="banner"> is rendered only on protected routes via Layout.js. Public pages lack a site-wide banner landmark. Screen reader users navigating to the login page cannot jump to a banner landmark.

Options:

  • Option A — Add a minimal header with role="banner" to public pages (logo + site name only, no user controls)
  • Option B — If omitting the header on public pages is a deliberate design decision, document this in the accessibility statement
⚠️ REMAINING #3 — useModalAriaHidden Not Applied to All Modal Components
WCAG 1.3.1 / 2.4.3 Severity: Medium Files: EventManagement.js, JobSeekerInterests.js

The useModalAriaHidden hook was created and applied to the three highest-priority components. EventManagement.js and JobSeekerInterests.js both contain Syncfusion DialogComponent usage with confirmOpen/confirmBulkDeleteOpen state but have not yet been updated.

// Add to EventManagement.js and JobSeekerInterests.js:
import useModalAriaHidden from '../../hooks/useModalAriaHidden';
// ...inside component function:
useModalAriaHidden(confirmOpen || confirmBulkDeleteOpen);
⚠️ REMAINING #4 — Syncfusion Grid Keyboard Trap Not Addressed
WCAG 2.1.2 No Keyboard Trap Severity: Medium Files: Dashboard.js, JobSeekerManagement.js (GridComponent with inline editing)

Syncfusion GridComponent in inline editing mode can trap keyboard focus. No Escape-key or Tab-exit handler was added for grid editing mode. This requires verification with an actual keyboard test (Tab into an editable grid cell, then attempt to exit with Escape or Tab past the grid).

Recommended approach: Add an actionComplete event handler on the GridComponent that listens for Escape key and calls grid.closeEdit(). Additionally, set editSettings={{ allowEditing: true, mode: 'Normal' }} (Normal mode is more keyboard-friendly than Batch mode for screen readers).
