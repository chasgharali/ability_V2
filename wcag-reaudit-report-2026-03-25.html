N-1 Login page now renders identical text as both <h1> and <h2> — duplicate heading announcement New High ▼
File: LoginPage.js:176, 213 WCAG: 1.3.1 Info and Relationships, 4.1.2 Introduced by: V-4.1 fix attempt

The fix for V-4.1 moved the <h1> to the top of the card correctly. However the old <h1> inside .login-form-container was downgraded to <h2> rather than removed. Both elements render the same text string and both are visible on screen simultaneously.

// LoginPage.js:176 — new h1 (correct position)
<h1 id="login-page-title" className="login-title">
    {userType === 'jobseeker' ? 'Job Seeker Login' : 'Company & Staff Login'}
</h1>

// LoginPage.js:213 — old heading, demoted to h2 but not removed
<h2 className="login-title">
    {userType === 'jobseeker' ? 'Job Seeker Login' : 'Company & Staff Login'}
</h2>
// Both render: "Job Seeker Login" — screen readers announce heading twice
Impact — Regression Screen reader users navigating by headings encounter the same page title twice, as a level-1 and then a level-2 heading. This inflates the heading structure and produces a confusing duplicate announcement every time the page loads or the user type tab is switched.
N-2 GlobalRouteObserver focuses #app-root then immediately overrides it with h1 focus in the same synchronous call New High ▼
File: FocusManager.js:193–197 WCAG: 2.4.3 Focus Order Introduced by: V-3.3 fix attempt

The fix for V-3.3 added appRoot.focus() before focusPageHeadingOrMain(). However, both calls are made synchronously within the same attemptFocus() function. The browser does not render between them, so the #app-root focus is immediately overridden by the <h1> focus in the same event-loop tick. The #app-root focus event may not even be dispatched. It never serves as an observable reset point for keyboard users.

// FocusManager.js:183-197 — same-tick double focus
const attemptFocus = () => {
    if (!document.title || !document.title.trim()) { ... }

    const appRoot = document.getElementById('app-root');
    if (appRoot) {
        appRoot.focus({ preventScroll: true });  // ← focus #1 (immediately lost)
    }
    focusPageHeadingOrMain();                    // ← focus #2 overrides #1 synchronously
};
// The spec requires #app-root to be a stable resting point so the next Tab
// lands on the skip link — but #app-root never holds focus long enough.
Impact — Regression The intent of focusing #app-root first (so Tab → skip link → main) is lost. After every route change, focus jumps directly to the <h1>, bypassing the skip link anchor entirely. Additionally, the spec's sequential protocol (document.title set → then focus) is still not enforced — the title check only applies when the title is completely empty.

Page Load, Title & Focus Protocol — 2 remaining

V-1.2 document.title → h1 focus sequence still not guaranteed High ▼
File: FocusManager.js:184–197 WCAG: 2.4.2 Page Titled Status since last audit: Unchanged

The new code checks if (!document.title || !document.title.trim()) and sets a fallback title from the h1 text. This only fires when the title is completely empty — it does not handle the race where <Helmet> is still writing the new title while focusPageHeadingOrMain() already runs. The focus call and the title update are still not sequenced.

// FocusManager.js:184-197
const attemptFocus = () => {
    if (!document.title || !document.title.trim()) {
        // Only handles empty title — does NOT wait for Helmet to finish writing
        document.title = `${fallbackHeading.textContent.trim()} - abilityconnect`;
    }
    appRoot.focus({ preventScroll: true });
    focusPageHeadingOrMain();   // ← no guarantee title is fully updated yet
};
Impact On fast navigation, the screen reader may announce the previous page's title (still in document.title) while simultaneously reading the new page's <h1>, producing a double-announce from two different page titles.
V-1.3 <h1> elements lack tabIndex={-1} as a React prop — patched by DOM mutation only High ▼
Files: EventDetail.js:100, LoginPage.js:176 (new h1) WCAG: 2.4.3 Focus Order Status since last audit: Unchanged

focusPageHeadingOrMain() still patches the attribute via heading.setAttribute('tabindex', '-1') at runtime (FocusManager.js:155). This DOM mutation is lost on any React re-render that replaces the element. Neither the new h1 in LoginPage.js (line 176) nor the h1 in EventDetail.js (line 100) carry tabIndex={-1} as a React prop.

// EventDetail.js:100 — still no tabIndex
<h1 style={{ textAlign: 'center', ... }}>{event.name}</h1>

// LoginPage.js:176 — newly introduced h1, also missing tabIndex
<h1 id="login-page-title" className="login-title">
    {userType === 'jobseeker' ? 'Job Seeker Login' : ...}
</h1>

// FocusManager.js:154-156 — fragile DOM patch, lost on re-render
heading.setAttribute('tabindex', '-1');
heading.focus();
Impact Programmatic .focus() on an element without tabindex="-1" silently fails in most browsers. Screen readers receive no focus event and no page-load signal.

Files: Auth.css:85, 283, 321, 427 WCAG: 2.4.7 Focus Visible Status since last audit: Unchanged — the .auth-card override was fixed (V-4.4) but component-level rules were not

Four component-level focus rules remain using :focus (fires on mouse click) instead of :focus-visible (keyboard only), and carry the wrong specification values.

Impact (1) Mouse users see distracting halos on every click on auth forms.
